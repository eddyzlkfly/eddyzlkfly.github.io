## Reconnaissance (Network Scanning)

I conducted a detailed port scan using **Nmap** on the target IP (**172.20.10.4**). My goal was to identify open ports, service versions, and any obvious misconfigurations using default scripts (`-sC`) and version detection (`-sV`).

```bash
nmap -sV -sC 172.20.10.4
```

### Scan Results & Critical Findings

![/assets/img/imgB2RL.png](/assets/img/imgB2RL.png)

The most promising lead is the **Anonymous FTP login** (FTP code 230). This is a common security oversight where a server allows anyone to log in with the username `anonymous` and no password.

## Exploiting Anonymous FTP

Based on the initial Nmap scan which indicated that **Anonymous FTP login** was enabled, I attempted to access the service using the username `anonymous` and leaving the password blank.

![/assets/img/imgB2RL(1).png](/assets/img/imgB2RL(1).png)

The login was successful, confirming a significant misconfiguration that allows unauthenticated users to access the server's file system.

Once connected, I navigated into the **`public`** directory. To ensure no data was missed, I executed the `ls -lah` command. The use of the **`-a`** flag was critical here, as it allowed me to view **hidden files** (those starting with a dot) that are typically excluded from standard directory listings.

This led to the discovery of a hidden file named **`.secret_note.txt`**.

To inspect the contents of the note, I used the **`get`** command to download it to my local attacker machine:

```bash
ftp> get .secret_note.txt
```

## Privilege Escalation Enumeration

To escalate my privileges from `librarian` to `root`, I began auditing the system's scheduled tasks. I inspected the system-wide configuration file located at **`/etc/crontab`**.

![/assets/img/imgB2RL(4).png](/assets/img/imgB2RL(4).png)

My enumeration revealed a highly interesting entry at the bottom of the file:

- `* * * * root /root/backup.sh`

### **Why is this significant?**

The Five-Star (`* * * * *`) syntax indicates that this specific job is executed **every minute, every day**. More importantly, it is executed by the **root** user.

This is a classic privilege escalation vector. Since the system automatically runs **`/root/backup.sh`** with the highest possible privileges every 60 seconds, any vulnerability within that script such as insecure file handling or the use of wildcards can be exploited to gain a root shell.

## Investigating the Backup Archive

To better understand the automated task running in the background, I explored the **`/var/backups`** directory. I identified a file named **`books.tar`** which appeared to be the archive generated by the `/root/backup.sh` script.

![/assets/img/imgB2RL(5).png](/assets/img/imgB2RL(5).png)

The file was world-readable, so I copied it to **books** directory where I had full permissions to investigate further.

Once copied, I used the **`tar`** command to extract the archive and see what was being backed up.

The extraction resulted in two files: **`guide.txt`** and **`rules.txt`**. Even though these files were empty, this was a major discovery. It confirmed that the **Root** user was actively archiving files from the **`/home/librarian/books`** folder.

This finding verified the link between my current user directory and the root-level cron job, highlighting a perfect opportunity for a **Tar Wildcard Injection** attack.

## Privilege Escalation via Tar Wildcard Injection

Knowing that a root-level cron job runs `tar *` in the `/home/librarian/books` directory, I performed a **Wildcard Injection** attack. This technique tricks the `tar` command into treating specially named files as system arguments.

![/assets/img/imgB2RL(6).png](/assets/img/imgB2RL(6).png)

### **Step-by-Step Execution**

- **Payload Creation:** Created a script named `shell.sh` to copy the bash binary to `/tmp/rootbash` and grant it **SUID** permissions.
- **Wildcard Injection:** Created two files named `-checkpoint=1` and `-checkpoint-action=exec=sh shell.sh` to trick the `tar` command into executing my payload as **root**.
- **Root Access:** Executed the newly created `/tmp/rootbash -p` binary to successfully spawn a persistent **root shell**.

## Gaining Root Access

After waiting a minute for the cron job to trigger, I checked the `/tmp` directory. A new binary named **`rootbash`** appeared with SUID permissions (`-rwsr-sr-x`).

By running the binary with the `-p` flag, I successfully spawned a root shell.

With the root shell established via the `rootbash` binary, I gained unrestricted access to the entire system. I navigated to the root home directory to retrieve the final objective: **`root.txt`**.
