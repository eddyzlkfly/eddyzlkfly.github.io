## Reconnaissance (Network Scanning)

The first step of the engagement was to identify the active services running on the target machine (**172.20.10.4**). I performed a comprehensive service discovery scan using **Nmap** to map out the attack surface.

```bash
nmap -sV -sC -A 172.20.10.4
```

### Scan Results & Analysis

![/assets/img/imgB2RF.png](/assets/img/imgB2RF.png)

After reviewing the results, I initially checked the **FTP (Port 21)** service. However, since I did not have any valid credentials at this stage and anonymous login was not explicitly permitted, I decided to pivot my focus to the web application running on **Port 80 (HTTP)**. 

The "Freshman Portal" appeared to be a more promising entry point for further manual exploration and directory brute-forcing.

## Web Exploration & Initial Access

Upon navigating to the target's IP address on port 80, I was presented with the **Freshman Portal** login page. 

![/assets/img/imgB2RF(1).png](/assets/img/imgB2RF(1).png)

Before proceeding with more complex exploitation techniques or automated brute-forcing, I decided to test for common default credentials.

I attempted to log in using the most common administrative pair:

```bash
admin:admin

```

Surprisingly, the application did not have robust password policies in place, and these credentials granted me immediate access to the administrative dashboard. 

This highlights a critical vulnerability where **default credentials** were left unchanged, providing an easy entry point for an attacker.

### Identifying the Upload Vector

After successful authentication, I was redirected to the **Assignment Upload** page (`upload.php`). 

![/assets/img/imgB2RF(2).png](/assets/img/imgB2RF(2).png)

The interface stated that only `.pdf` or `.docx` formats were recommended for submission.

From an attacker's perspective, this functionality is a high-interest target for **Unrestricted File Upload** vulnerabilities. My immediate strategy was to attempt to bypass any client-side or server-side filters to upload a **PHP Reverse Shell**. 

If successful, this would allow me to execute arbitrary commands on the server and establish a persistent connection back to my machine.

### Payload Crafting & Bypass Techniques

To bypass the application's security filters, I crafted a specialized payload named **`shell.pdf.php`**. I implemented two primary bypass techniques to ensure successful upload and execution:

![/assets/img/imgB2RF(3).png](/assets/img/imgB2RF(3).png)

- **Double Extension Bypass (`.pdf.php`):** The web server's filter likely checks for the presence of the `.pdf` extension to validate the file type. By using a double extension, I satisfy this initial check while ensuring the Apache web server processes the file based on the final `.php` extension, allowing for server-side code execution.
- **Magic Bytes / File Signature Spoofing:** I inserted the header **`%PDF-1.4`** at the very beginning of the file. This is known as the "Magic Byte" or file signature. Some servers verify the file type by reading the first few bytes of the content rather than just the extension. This header tricks the server's backend into believing the file is a legitimate PDF document.

Once this file is uploaded and accessed via the browser, the server will execute the bash command and connect back to my listener, providing me with remote terminal access.

### Executing the Reverse Shell

After bypassing the file upload restrictions, the application confirmed a successful submission and provided a direct path to the uploaded file: `/uploads/shell.pdf.php`.

![/assets/img/imgB2RF(4).png](/assets/img/imgB2RF(4).png)

To trigger the payload, I first set up a **Netcat** listener on my attacker machine to wait for the incoming connection on the specified port:

![/assets/img/imgB2RF(5).png](/assets/img/imgB2RF(5).png)

By navigating to the provided URL, the web server (Apache) processed the file as a PHP script. This executed my encoded bash command, which initiated a callback to my machine.

The listener successfully captured the connection, granting me a remote shell.

## Authenticating via FTP

Armed with the credentials found in the development notes (`freshman:freshman123`), I attempted to log in to the **FTP service (Port 21)**. 

![/assets/img/imgB2RF(8).png](/assets/img/imgB2RF(8).png)

The connection was successful, providing me with direct access to the home directory of the **`freshman`** user.

After entering the FTP environment, I executed the `ls` command and immediately located the first target file: **`user.txt`**.

![/assets/img/imgB2RF(9).png](/assets/img/imgB2RF(9).png)

### **Why "get" instead of "cat"?**

In a standard FTP session, you cannot use the `cat` command to view the contents of a file directly on the server. FTP (File Transfer Protocol) is strictly designed for **transferring** files, not for interactive file manipulation. To read the flag, I had to download it to my local machine first using the **`get`** command:

```bash
ftp> get user.txt
```

## Data Decoding & Flag Capture

Once the **`user.txt`** file was successfully transferred to my local machine, I used the `cat` command to inspect its content. 

![/assets/img/imgB2RF(10).png](/assets/img/imgB2RF(10).png)

Instead of a plain-text flag, the file contained a string of alphanumeric characters ending with an `=` sign:

```bash
aGFjazEwezNhc3lfcDNhc3lfMW4xdDFhbF9hY2MzN9Cg==
```

Based on the character set and the specific padding, I identified this as **Base64 encoding**. In many CTF challenges, flags are encoded to prevent them from being easily "grepped" or read without a basic level of analysis.

## Privilege Escalation Discovery

With the credentials obtained from the previous enumeration phase, I returned to my active shell and used the **`su`** (Substitute User) command to switch from the restricted `www-data` service account to the **`freshman`** user account.

![/assets/img/imgB2RF(11).png](/assets/img/imgB2RF(11).png)

I successfully authenticated using the password `freshman123`.

The first command I execute after gaining access to a new user account is **`sudo -l`**

### **What is `sudo -l`?**

In simple terms, this command asks the system: *"What administrative powers does this user have?"*. It lists all the commands that the current user is allowed to run as **root** (or another user) using the `sudo` prefix. It is a critical step in identifying "low-hanging fruit" for privilege escalation.

The output of `sudo -l` revealed a significant misconfiguration:
**`(ALL) NOPASSWD: /usr/bin/find`**

This indicates that the user `freshman` can run the **`find`** binary as any user (including **root**) without being prompted for a password.

## Final Privilege Escalation & Root Capture

Since I discovered that the `freshman` user can execute **`/usr/bin/find`** with Root privileges (via `sudo`) without a password, I leveraged the **`-exec`** flag to bypass file system restrictions.

![/assets/img/imgB2RF(12).png](/assets/img/imgB2RF(12).png)

The `-exec` flag allows the `find` command to run an external system command on any file it locates. By pointing `find` to a file that only Root can access and executing `cat`, I can read sensitive files that are otherwise protected.

Execution of the command returned another Base64 encoded string:

```bash
aGFjazEwe3Iwb3RfcHIxdjNzY192MWFfczB1ZDBfZjFuZF93MDB0fQo=
```

Following the same procedure as the user flag, I decoded the string using the `base64` utility

`hack10{r00t_pr1v3sc_v1a_s0ud0_f1nd_w00t}`
