## **“Fresh”**

I just created my custom plugin for my reporting page.

## Solution

### Reconnaissance with Nmap

The first thing we do is run an **Nmap scan** against the target machine at `172.20.10.4`.

```html
nmap -sCV 172.20.10.4 --vv
```

Results:

![/assets/img/imgAK26B2R.png](/assets/img/imgAK26B2R.png)

The scan reveals two open ports:

- **Port 22** — SSH (OpenSSH 8.9p1 on Ubuntu)
- **Port 80** — HTTP (Apache 2.4.52 running **WordPress 6.9.4**)

The page title shows **"Ali's Reports"**, which already tells us the WordPress site belongs to a user named **Ali**. This is useful information we'll come back to.

### Brute Forcing WordPress Credentials via XMLRPC

Since we know the target is running WordPress and there's a user called **Ali**, we attempt a password attack against the **XMLRPC endpoint,** a common WordPress attack vector.

![/assets/img/imgAK26B2R(1).png](/assets/img/imgAK26B2R(1).png)

The brute force is successful:

`[SUCCESS] - ali / michael1
[SUCCESS] - Ali / michael1`

Valid credentials found:

- **Username:** ali
- **Password:** michael1

We now have access to the WordPress admin panel.

### Discovering the Vulnerable Custom Plugin

After logging in, we navigate to the **"My Reports"** page in the WordPress editor. We can see it uses a custom shortcode called `[include_report]` with a `path` parameter pointing to various report files:

![/assets/img/imgAK26B2R(2).png](/assets/img/imgAK26B2R(2).png)

This plugin reads files from the server based on the `path` value and if there's no sanitization, we can manipulate this path to read **any file on the system**. This is a classic **Local File Inclusion (LFI)** vulnerability.

### Exploiting LFI to Steal the SSH Private Key

After a few enumeration, we craft a malicious path using directory traversal to escape the WordPress root and reach the user's SSH directory:

**LFI payload:**

```bash
../../../../../home/ali/.ssh/id_rsa
```

We inject this into the shortcode's `path` parameter and visit the page. The plugin blindly reads the file and renders its content which exposing **Ali's private SSH key** directly on the page:

![/assets/img/imgAK26B2R(3).png](/assets/img/imgAK26B2R(3).png)

The server had no input validation whatsoever on the `path` parameter, allowing us to traverse outside the intended directory and read sensitive files.

### SSH Login as Ali

We save the stolen private key to a local file called `ali_key`, set the correct permissions, and connect via SSH:

![/assets/img/imgAK26B2R(4).png](/assets/img/imgAK26B2R(4).png)

We're in! The system confirms we're logged in as **ali** on the machine named **Fresh**, running Ubuntu 22.04 LTS.

### Capturing the User Flag

Once inside, we run basic enumeration commands to confirm our identity and look around:

![/assets/img/imgAK26B2R(5).png](/assets/img/imgAK26B2R(5).png)

## 🚩 User Flag

`AK26{R4HHHH_1N1TI4L_4CC3SS_V1A_SSH!!!}`

### Privilege Escalation: Finding SUID Binaries

Now we need to escalate from **ali** to **root**. We search for binaries with the **SUID bit** set, which allows them to run with elevated (root-level) privileges:

![/assets/img/imgAK26B2R(6).png](/assets/img/imgAK26B2R(6).png)

Among the usual system binaries, one stands out **`/usr/bin/vim.basic`** is highlighted in the results. Vim having a SUID bit means it can be exploited to execute commands as root.

### Exploiting Vim SUID for Root Shell

We use the following payload to abuse Vim's built-in Python3 support and spawn a root shell:

```bash
/usr/bin/vim.basic -c ':py3 import os; os.setuid(0); os.execl("/bin/sh", "sh", "-c", "reset; exec sh")'
```

Here's what this does:

- `:py3` — executes Python3 code from within Vim
- `os.setuid(0)` — sets our user ID to **0 (root)**
- `os.execl("/bin/sh", ...)` — drops us into a **root shell**

Running `whoami` confirms we are now **root**.

![/assets/img/imgAK26B2R(7).png](/assets/img/imgAK26B2R(7).png)

### Capturing the Root Flag

With root access, we navigate to `/root/` and read the final flag:
