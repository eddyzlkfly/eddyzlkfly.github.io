## **"Not Meant For You"**

A newly deployed internal portal was found during routine testing.

Everything appears to be functioning as expected… login works, profiles load, nothing unusual at first glance.

But sometimes, what a system shows you isn’t the full story.

With development still in progress, there may be small oversights. Things left behind or assumptions that shouldn’t have been trusted.

Can you uncover what was never meant to be seen?

**Hint 1:** Not all logic belongs on the server.

**Hint 2:** What happens if you look beyond your own data?

**Hint 3:** Numbers can open doors… if no one is checking them.

## Solutions

We're greeted with the **USIM Internal Portal** homepage, described as an employee profile system with a Login button. Nothing suspicious on the surface — but let's see what's hiding underneath.

![/assets/img/imgAK26Web.png](/assets/img/imgAK26Web.png)

Clicking Login brings us to a standard login form with Username and Password fields. The question is what credentials do we use?

![/assets/img/imgAK26Web(1).png](/assets/img/imgAK26Web(1).png)

This is where Hint 1 kicks in: *"Not all logic belongs on the server."*

Opening **Developer Tools → Sources → app.js**, we find this sitting in plain sight:

![/assets/img/imgAK26Web(2).png](/assets/img/imgAK26Web(2).png)

```jsx
// TODO: remove before production
// temporary test account
const testUser = {
    username: "user",
    password: "user123"
};
```

The developer left hardcoded credentials inside **client-side JavaScript** is a critical mistake. Any visitor can open DevTools and read this. We use these credentials to log in successfully.

After logging in, we notice the URL changes to:

```
/profile?id=1
```

![/assets/img/imgAK26Web(3).png](/assets/img/imgAK26Web(3).png)

This shows the profile of **Daniel Carter**, an IT Support staff. This appears to be our own account. But Hint 2 says: *"What happens if you look beyond your own data?"* So let's try changing that number.

We manually change the URL to `?id=2` and are now looking at someone else's profile which is **Sarah Mitchell**, HR Assistant.

The server **never checked** whether we're authorized to view this profile. This is a textbook **IDOR (Insecure Direct Object Reference)** vulnerability. We can access anyone's data just by changing a number in the URL.

![/assets/img/imgAK26Web(4).png](/assets/img/imgAK26Web(4).png)

Continuing to `?id=3`, we find **James Walker**, Finance Officer.

The pattern is clear, there is absolutely no access control here. Any logged-in user can enumerate through all profiles simply by guessing ID numbers.

![/assets/img/imgAK26Web(5).png](/assets/img/imgAK26Web(5).png)

Trying `?id=4` returns **"User not found"**. But don't stop here! Opening the **Elements tab** in DevTools reveals something interesting in the HTML source:

```html
<!-- TODO: review user ID allocation -->
<!-- special accounts may use identifiers outside normal patterns -->
```

![/assets/img/imgAK26Web(6).png](/assets/img/imgAK26Web(6).png)

Hint 3 says: *"Numbers can open doors… if no one is checking them."* This comment suggests there are accounts with unusual IDs, perhaps something far outside the normal range like **999**.

Navigating to `?id=999` reveals an **Admin - Access Restricted** page. We can't get in, but inspecting the source code once again uncovers a hidden comment:

```html
<!-- TODO: remove backup before deployment -->
<!-- backup: /backup/admin.txt -->
```

![/assets/img/imgAK26Web(7).png](/assets/img/imgAK26Web(7).png)

The developer exposed a backup file path directly in the HTML. Anyone who opens DevTools can see this a serious information disclosure vulnerability.

We navigate directly to `/backup/admin.txt` and find this message:

> *"There's nothing useful here... But maybe logs reveal more: /logs/system.log"*
> 

![/assets/img/imgAK26Web(8).png](/assets/img/imgAK26Web(8).png)

The backup file itself hints at yet another exposed path. We follow the breadcrumb.

Finally, navigating to `/logs/system.log` reveals:

![/assets/img/imgAK26Web(9).png](/assets/img/imgAK26Web(9).png)

The flag was stored inside a **debug log file that's publicly accessible,** no authentication required. This is another severe vulnerability, as system logs should never be exposed to unauthenticated users.
